# DeepSentinel
Nisha
DeepSentinel is a Windows-focused insider-threat monitoring and response system. It collects endpoint activity from agent machines, sends events to a Flask server, scores risky behavior, and shows alerts in a live security dashboard.

The project is designed for demonstrations, academic work, and controlled lab environments. It monitors user activity such as file operations, USB usage, login/logoff events, clipboard activity, email activity, process activity, honeypot triggers, and multi-machine endpoint status.

## Key Features

- Live admin dashboard for endpoint activity, alerts, incidents, users, and risk scores.
- Windows endpoint agent for file, USB, process, clipboard, email, and logon/logoff monitoring.
- After-hours login and logoff detection outside office hours, currently 09:00 to 17:00.
- USB device detection and large USB file-transfer detection.
- Risk scoring with local heuristics and model-assisted scoring.
- Email risk classification and admin approval workflow.
- Honeypot file, credential, and service trap support.
- User blocking, watch mode, and isolation workflows.
- Multi-LAN endpoint visibility for tracking multiple computers.
- Audit trail and incident/report generation support.
- Real-time notification hooks and dashboard alert tables.

## Recent Fixes

The latest update fixes three important detection and dashboard issues:

1. After-hours logoff detection
   - Previously, Windows logoff events were sent as `event_type=logon` with `action=user_logoff`, so the server treated them like normal login events.
   - The server now recognizes logoff actions correctly.
   - Alerts are generated for both `after_hours_login` and `after_hours_logoff`.

2. Large USB file-transfer detection
   - USB plug-in events were detected, but copying a large file to the USB drive was not.
   - The agent now snapshots removable-drive files and detects new or modified large files.
   - Large USB transfer events are emitted as `event_type=usb` and `action=large_file_transfer`.
   - The dashboard now shows these as critical alerts.

3. Dashboard time correctness
   - Event timestamps used a mix of UTC and local time.
   - Agent timestamps and server received/alert timestamps now use local system time consistently.
   - This fixes incorrect times shown in dashboard tables and alerts.

## Configuration

Main server thresholds are in:

```text
config.json
FS/insepticon/config.json
```

Important settings:

```json
{
  "thresholds": {
    "files_created_today": 12,
    "http_requests_today": 500,
    "bytes_downloaded_today": 104857600,
    "large_usb_transfer_bytes": 52428800
  },
  "risk_display_threshold": 0.5
}
```

`large_usb_transfer_bytes` controls when a copied USB file becomes a critical transfer alert. The current default is 50 MB.

Agent settings are in `FS/insepticon/file_agent.py` and `config.json` generated by the agent. Important values include:

- `server_url`: server receiver URL, usually `http://127.0.0.1:5000/receive_log`
- `watch_paths`: folders monitored for file activity
- `max_upload_risk_size_mb`: large-file threshold used by the agent
- `enable_process_monitor`: process monitoring switch
- `enable_outlook_monitor`: Outlook monitoring switch
- `enable_imap_monitor`: IMAP monitoring switch

### Secrets

Do not hardcode API keys in source files. The report generator reads the Google Gemini key from the `GEMINI_API_KEY` environment variable.

PowerShell:

```powershell
$env:GEMINI_API_KEY="your-google-gemini-api-key"
```

Bash:

```bash
export GEMINI_API_KEY="your-google-gemini-api-key"
```

You can also copy `.env.example` to `.env` for local development, but never commit `.env`.

## Running Locally

From the project root:

```bash
cd "DEEPSENTINEL/DEEPSENTINEL/FS (1)/FS"
```

Start the server:

```bash
python FS/insepticon/server.py
```

Open the dashboard:

```text
http://127.0.0.1:5000/dashboard
```

Start the endpoint agent in another terminal:

```bash
python FS/insepticon/file_agent.py
```

For logon/logoff monitoring, run the agent terminal as Administrator. Without admin rights, Windows Security Event Log monitoring may be skipped.

## Detection Coverage

| Detection area | Event type | Alert behavior |
| --- | --- | --- |
| After-hours login | `logon` / `user_logon` | Creates `after_hours_login` alert |
| After-hours logoff | `logon` / `user_logoff` | Creates `after_hours_logoff` alert |
| USB plug-in | `usb` / `plugged` or `device_connected` | Logged and risk-scored |
| USB removal | `usb` / `unplugged` or `device_removed` | Logged |
| Large USB transfer | `usb` / `large_file_transfer` | Creates critical `large_usb_transfer` alert |
| File activity | `file` | Created, modified, deleted, moved events |
| Sensitive keywords | `file` | Raises file risk score |
| Process activity | `process` | Suspicious process start/end events |
| Clipboard activity | `clipboard` | Sensitive copied text detection |
| Email activity | `outlook`, `imap`, `email` | ML classification and approval queue |
| Honeypot access | honeypot events | Critical incident and response workflow |

## Runtime Data and GitHub Notes

The project writes runtime data into `data/` and `FS/insepticon/data/`, including event logs, alerts, debug logs, screenshots, generated reports, encrypted local logs, and server logs.

A `.gitignore` file is included to keep generated files, local secrets, logs, model artifacts, virtual environments, and runtime data out of Git.

Before pushing to GitHub, do not commit:

- `data/*.jsonl`
- `data/*.log`
- `FS/insepticon/data/*.jsonl`
- `FS/insepticon/data/*.log`
- `secure_local_log.bin`
- `secure_local_log.bin.jsonl`
- `agent_key.key`
- `server.log`
- `__pycache__/`
- virtual environments such as `.venv/` and `deepenv/`
- generated screenshots, recordings, reports, and quarantine files

These files can contain local usernames, machine names, paths, screenshots, encrypted logs, or sensitive test data.

If these files were already added to Git before `.gitignore` existed, remove them from Git tracking without deleting your local copies:

```bash
git rm -r --cached data FS/insepticon/data FS/insepticon/logs FS/insepticon/quarantine
git rm --cached agent_key.key secure_local_log.bin secure_local_log.bin.jsonl server.log threat_log.txt
git rm -r --cached FS/insepticon/models FS/insepticon/deepenv .venv
git add .gitignore README.md
git status
```

Only commit source code, configuration templates, documentation, and small sample files that do not contain private data.

Recommended upload checklist:

1. Confirm `.gitignore` is present.
2. Run `git status` and review every staged file.
3. Do not upload logs, screenshots, generated reports, keys, virtual environments, or trained model binaries unless you intentionally want them public.
4. Add a `LICENSE` file before publishing.
5. Add sanitized sample data only if needed for demonstration.

If GitHub secret scanning reports a leaked key:

1. Revoke or rotate the key in the provider dashboard immediately.
2. Remove the secret from the current code.
3. Purge the secret from Git history before pushing again, or create a fresh clean repository.
4. Re-scan the repository for the leaked pattern.

## Security Notice

DeepSentinel is a monitoring and response prototype. Use it only on systems where you have permission to monitor activity. Logon/logoff monitoring may require Administrator access because it reads Windows Security Event Logs. Some response actions, such as blocking or isolation, can affect normal user activity and should be tested in a controlled environment first.
